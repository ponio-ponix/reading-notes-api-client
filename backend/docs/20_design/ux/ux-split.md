# スマホUI と WebUI の役割分担（UXアーキテクチャ）

本プロダクトは **「引用の高速キャプチャ」** と **「知識・思考の深い整理」** の両立を目的とする。  
そのため、スマホUI と WebUI は **同じUIの縮小版／拡大版ではなく、用途が根本的に異なる** という前提で設計する。

---

## 1. 前提：読書 × メモ行動モデル

行動は以下の2つに明確に分かれる。

### **スマホ（インプット端末）**
- 移動中に読む
- 気になった箇所を即座に保存
- カメラで引用キャプチャ
- Kindleのハイライト追加
- “後で整理するためのインボックス”として使う

→ **目的：高速キャプチャ（3〜5秒以内）**

---

### **Web（PC：アウトプット端末）**
- 引用の整理
- 思考ノートの作成
- タグ付け / 検索
- 本を横断した概念整理
- 読書会の発表内容まとめ

→ **目的：深い整理・編集・分析**

---

## 2. スマホUIの設計原則：キャプチャ専用機に最適化

スマホUIは “編集させない”。  
必要なのは以下の数ステップのみ。

- 本を選ぶ（自動推測 or 最近使った本が上位に出る）
- ページ番号入力（可能なら自動推定 or スクロール式）
- 引用文の入力  
  - テキスト入力  
  - カメラOCR  
  - コピペ
- メモ（任意）
- 保存（1タップ）

**目的：とにかく高速にインプットを突っ込むこと。**

---

## 3. WebUIの設計原則：知識構造の母艦

WebUIはスマホの真逆で、以下の機能に最適化する。

- 本ごとの引用一覧（並べ替え / フィルタ）
- タグ管理
- 引用横断検索
- 思考ノート（メモを育てる場所）
- 読書会ステートメント作成
- 引用統合ビュー（Insight生成）
- Markdown出力

**目的：インプットを知識に変換する “思考プラットフォーム”。**

---

## 4. スマホとWebの関係性（重要）

| 役割 | スマホ | Web |
|------|--------|------|
| 入力速度 | 最速（3〜5秒） | 遅いが構造化に強い |
| 編集 | 基本なし | すべてここで行う |
| 思考展開 | 不向き | 主戦場 |
| 情報量 | 最小 | 最大 |
| 主目的 | キャプチャ | 整理・活用 |

**スマホ = インボックス**  
**Web = ナレッジベース**

---


## 5. 本プロダクトの核となる思想

- “スマホに整理を求めない”
- “PCに高速性を求めない”
- “用途の分離こそ設計の本質”
- “優秀層のワークフローと完全一致するUIを提供する”

この思想をもとに、全ての画面設計・API設計・データ構造を決定する。

---

## 6. 今後の設計タスク（ドラフト）

- スマホの「引用キャプチャ動線」を秒単位で最適化する
- Webの「引用一覧 → ノート統合」UIのワイヤーフレーム作成
- 引用データモデル（Quote）を確定する  
  - book_id  
  - page  
  - text  
  - memo  
  - tags  
  - created_at / updated_at
- インボックス（未整理状態）の定義
- “整理済み” の定義（＝ナレッジベース側）

---

## 結論

**スマホはキャプチャ装置、Webは思考装置。  
この分離を設計の心臓に据える。**
